### Backend Layer: Convex

Convex is the complete backend platform providing:

<Tabs>
  <Tab title="Database">
    **Reactive Database**
    
    Convex provides a fully managed database with:
    - Automatic real-time subscriptions
    - ACID transactions
    - TypeScript schema validation
    - Automatic indexes
    - Vector storage for embeddings
    
    ```typescript convex/schema.ts
    export default defineSchema({
      persons: defineTable({
        name: v.string(),
        slug: v.string(),
        status: v.union(
          v.literal("pending"),
          v.literal("processing"),
          v.literal("ready"),
          v.literal("failed")
        ),
        createdAt: v.number(),
      }).index("by_slug", ["slug"]),
      
      chunks: defineTable({
        personId: v.id("persons"),
        text: v.string(),
        embedding: v.array(v.float64()),
        citation: v.string(),
      }).index("by_person", ["personId"]),
    });
    ```
  </Tab>
  
  <Tab title="Functions">
    **Three Function Types**
    
    <Steps>
      <Step title="Queries (Read)">
        Fast, cached reads with automatic subscriptions
        ```typescript
        export const getPerson = query({
          args: { personId: v.id("persons") },
          handler: async (ctx, args) => {
            return ctx.db.get(args.personId);
          },
        });
        ```
      </Step>
      
      <Step title="Mutations (Write)">
        Transactional writes that trigger reactive updates
        ```typescript
        export const createPerson = mutation({
          args: { name: v.string() },
          handler: async (ctx, args) => {
            return ctx.db.insert("persons", {
              name: args.name,
              status: "pending",
              createdAt: Date.now(),
            });
          },
        });
        ```
      </Step>
      
      <Step title="Actions (External)">
        Long-running operations that call external APIs
        ```typescript
        export const runIngestion = action({
          args: { personId: v.id("persons") },
          handler: async (ctx, args) => {
            // Call OpenAI, Exa, fetch URLs
            const result = await externalAPI();
            // Write back via mutations
            await ctx.runMutation(internal.pipeline.save, {
              data: result
            });
          },
        });
        ```
      </Step>
    </Steps>
  </Tab>
  
  <Tab title="Scheduling">
    **Built-in Job Scheduler**
    
    Convex includes a scheduler for background jobs:
    ```typescript
    // Schedule ingestion pipeline
    await ctx.scheduler.runAfter(
      0, // milliseconds delay
      api.pipeline.runIngestion,
      { personId, phase: "discover" }
    );
    ```
    
    This enables:
    - Pipeline phase chaining
    - Automatic retries on transient failures
    - Resumable long-running operations
  </Tab>
</Tabs>

## Data Flow

### User Request Flow

<Steps>
  <Step title="User enters a name">
    Next.js page calls Convex mutation:
    ```typescript
    const { personId } = await createPerson({ name });
    ```
  </Step>
  
  <Step title="Bootstrap & Schedule">
    Mutation creates database records and schedules ingestion:
    ```typescript
    const personId = await ctx.db.insert("persons", {...});
    await ctx.scheduler.runAfter(0, api.pipeline.runIngestion);
    ```
  </Step>
  
  <Step title="Pipeline Execution">
    Action runs through 5 phases (see [Data Pipeline](/platform/data-pipeline)):
    - Discover → Extract → Stage → Embed → Publish
  </Step>
  
  <Step title="Real-time Updates">
    Frontend automatically receives updates via Convex subscriptions:
    ```typescript
    const result = useQuery(api.persons.getPerson, { personId });
    // Re-renders automatically when status changes
    ```
  </Step>
</Steps>

### Chat Flow (RAG)

```
User Question
     |
     v
1. Embed query → OpenAI embeddings
     |
     v
2. Vector search → Cosine similarity across chunks
     |
     v
3. Retrieve top 8 chunks → Stage-scoped or fallback
     |
     v
4. Build context → Evidence + voice samples + stage info
     |
     v
5. Generate response → OpenAI GPT-5.2
     |
     v
6. Return answer + citations
```

See [AI Models](/platform/ai-models) for detailed RAG implementation.

## Database Schema

<Accordion title="Core Tables">
  **persons** - Biography subjects
  ```typescript
  {
    name: string,
    slug: string,
    status: "pending" | "processing" | "ready" | "failed",
    birthDate?: string,
    createdAt: number,
    updatedAt: number
  }
  ```
  
  **jobs** - Pipeline tracking
  ```typescript
  {
    personId: Id<"persons">,
    phase: "discover" | "extract" | "stage" | "embed" | "publish",
    status: "queued" | "running" | "done" | "failed",
    progress: number,
    error?: string
  }
  ```
  
  **sources** - Discovered content
  ```typescript
  {
    personId: Id<"persons">,
    url: string,
    title: string,
    type: "article" | "video" | "post" | "interview" | "other",
    rawText?: string,
    transcriptText?: string,
    qualityScore: number
  }
  ```
  
  **stages** - Life stages
  ```typescript
  {
    personId: Id<"persons">,
    order: number,
    title: string,
    ageStart: number,
    ageEnd: number,
    eraSummary: string,
    worldviewSummary: string,
    turningPoints: string[],
    confidence: number
  }
  ```
  
  **chunks** - Embedded text for RAG
  ```typescript
  {
    personId: Id<"persons">,
    sourceId: Id<"sources">,
    stageId?: Id<"stages">,
    text: string,
    embedding: number[], // 1536-dim vector
    citation: string // JSON
  }
  ```
</Accordion>

## Key Design Decisions

<AccordionGroup>
  <Accordion title="Why Convex over traditional backend?">
    **Benefits:**
    - No API routes to write - direct function calls
    - Automatic real-time subscriptions
    - Built-in scheduling for pipeline phases
    - Type-safe end-to-end
    - Integrated database + functions + jobs
    
    **Trade-offs:**
    - Vendor lock-in (but migration path exists)
    - Learning curve for team
    - Less ecosystem compared to Express/Postgres
  </Accordion>
  
  <Accordion title="Why store embeddings in Convex vs dedicated vector DB?">
    **Rationale:**
    - Keeps all data in one system
    - Cosine similarity is computed in-memory (fast enough for under 10k chunks/person)
    - Avoids another service to manage
    - Can migrate to Pinecone/Weaviate if scale requires
    
    Current implementation: `convex/pipeline.ts:140-152`
  </Accordion>
  
  <Accordion title="Why phase-based pipeline vs single job?">
    **Advantages:**
    - Resumable: Each phase persists results before chaining
    - Debuggable: Can inspect state between phases
    - Timeout-resistant: Phases schedule next phase, so timeouts don't lose work
    - Flexible: Can re-run specific phases
    
    See [Data Pipeline](/platform/data-pipeline) for implementation details.
  </Accordion>
</AccordionGroup>

## Performance Characteristics

| Operation | Latency | Notes |
|-----------|---------|-------|
| Query (cached) | Under 50ms | Convex edge cache |
| Mutation | 100-300ms | Database write + broadcast |
| Action (simple) | 200-500ms | External API call |
| Full ingestion | 3-8 minutes | All 5 phases for new person |
| Embed phase | 30-90 seconds | Batch embedding 100-500 chunks |
| Chat response | 1-3 seconds | Retrieval + GPT-5.2 generation |

<Warning>
The **embed** phase can consume significant OpenAI tokens. We batch 20 texts per API call and limit to 10 chunks per source to control costs.

See `convex/pipeline.ts:1224-1326` for implementation.
</Warning>
