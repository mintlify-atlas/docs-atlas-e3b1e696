# The 5-Phase Pipeline

LORE Timeline uses a sophisticated AI-powered pipeline to transform a simple name into a rich, interactive biography. Every phase builds on the previous one, creating a comprehensive knowledge graph from raw web data.

## Architecture Overview

The pipeline is built with **resilience** and **resumability** in mind:

- Each phase completes fully before starting the next
- Progress is saved after every phase, so failures don't lose work
- Phases can be resumed individually if a timeout occurs
- All data is stored in Convex's real-time database

```typescript
// From convex/pipeline.ts - Phase orchestration
export const runIngestion = action({
  handler: async (ctx, args) => {
    const startPhase = args.phase ?? "discover";
    
    // Each phase completes and chains to the next
    if (startPhase === "discover") {
      // ... do discovery work ...
      await ctx.scheduler.runAfter(0, api.pipeline.runIngestion, {
        personId: args.personId,
        phase: "extract",
      });
    }
    // ... continue through all phases ...
  },
});
```

## Phase 1: Discover

**Goal:** Find high-quality biographical sources across the web

### What Happens

<Steps>
  <Step title="Domain detection">
    LORE uses AI to infer the person's domain (musician, athlete, entrepreneur, scientist, etc.) and tailor search strategies accordingly.
  </Step>
  
  <Step title="Query generation">
    GPT-4 generates **8-12 targeted search queries** specific to that person:
    - 2-3 general biography/profile searches
    - 2-3 domain-specific searches (e.g., "discography" for musicians)
    - 2-3 mainstream video searches (interviews, talks)
    - 2-3 underground/rare video searches (before they were famous)
    - 1 historical search (filtered by date for older content)
  </Step>
  
  <Step title="Execute searches">
    All queries are executed via **Exa API**, which provides semantic search across the web and YouTube:
    - Web searches return 8 results each
    - Video searches return 5 results each
    - Results are deduplicated by URL
  </Step>
  
  <Step title="Vet sources">
    AI reviews all discovered sources to confirm they're about the RIGHT person (not someone with the same name):
    - Analyzes title, URL, and snippet
    - Filters out tangential mentions
    - Only keeps sources clearly about the target person
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - Query generation
const queryPlanJson = await generateJson(
  `You are a research agent. Given a person's name, generate a comprehensive 
  set of search queries to discover biographical sources about them.

  actor, entrepreneur, activist, author, etc. Tailor queries to their likely domain.
  
  Output a JSON object: { "queries": [...] }
  
  Each query object has:
  - "query": the search string (use quotes around the person's name)
  - "type": "web" or "video"
  - "dateEnd": optional ISO date to find older content

  JSON.stringify({ personName: person.name })
);
```

### Output

A list of **20-50 candidate sources** with:
- URL
- Title
- Type (article, video, interview, post)
- Published date (when available)
- Snippet preview

<Tip>
**Fallback Strategy:** If no sources are found (e.g., API failure), LORE falls back to curated searches like Wikipedia, YouTube search URLs, and DuckDuckGo query links.
</Tip>

## Phase 2: Extract

**Goal:** Pull full text content and video transcripts from all discovered sources

### What Happens

<Steps>
  <Step title="Detect content type">
    For each source URL, LORE determines the extraction strategy:
    - **YouTube videos** → Fetch transcript via YouTube's caption API
    - **Web articles** → Fetch HTML and strip to plain text
    - **Search result URLs** → Use metadata only (title + snippet)
  </Step>
  
  <Step title="YouTube transcript extraction">
    For YouTube videos:
    - Fetch the video page HTML
    - Extract caption track URLs from embedded JSON
    - Download transcript XML (prefer English, fallback to first available)
    - Parse and clean transcript text (up to 30,000 characters)
  </Step>
  
  <Step title="Web scraping">
    For articles and web pages:
    - Fetch HTML with a user agent header
    - Strip all HTML tags, scripts, and styles
    - Extract image URLs (for person photos)
    - Keep first 20,000 characters of text
  </Step>
  
  <Step title="Image extraction">
    From HTML content, LORE identifies relevant person photos:
    - Filters by alt text containing the person's name
    - Skips icons, logos, tracking pixels, and site chrome
    - Keeps images larger than 150x100 pixels
    - Returns up to 4 images per source
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - YouTube transcript fetching
async function fetchYoutubeTranscript(videoId: string): Promise<string> {
  // Step 1: Fetch video page to extract caption tracks
  const pageResponse = await fetch(
    `https://www.youtube.com/watch?v=${videoId}`
  );
  const pageHtml = await pageResponse.text();
  
  // Extract the captions JSON from page source
  const captionMatch = pageHtml.match(/"captionTracks":\s*(\[.*?\])/);
  const captionTracks = JSON.parse(captionMatch[1]);
  
  // Prefer English, fallback to first available
  const track = captionTracks.find(t => t.languageCode === "en") 
    ?? captionTracks[0];
  
  // Step 2: Fetch transcript XML
  const captionResponse = await fetch(track.baseUrl + "&fmt=srv3");
  const captionXml = await captionResponse.text();
  
  // Parse and clean text segments
  const textSegments = [];
  const segmentRegex = /<p[^>]*>(.*?)<\/p>/gi;
  let match;
  while ((match = segmentRegex.exec(captionXml)) !== null) {
    textSegments.push(cleanText(match[1]));
  }
  
  return textSegments.join(" ").slice(0, 30000);
}
```

### Output

Each source now has:
- **rawText** - Full article/page content (up to 20k chars)
- **transcriptText** - YouTube transcript (up to 30k chars)
- **imageUrls** - Array of relevant person photos

<Info>
**Resume Capability:** Extract is the longest phase. If it times out, it skips sources that already have text extracted, so you don't lose progress.
</Info>

## Phase 3: Stage

**Goal:** Divide the person's life into meaningful eras (stages) with summaries and turning points

### What Happens

<Steps>
  <Step title="Source digest">
    LORE builds a summary of all sources:
    - First 30 sources (or all if fewer)
    - Title, URL, type, and 1000-character text sample from each
    - Passed to GPT-4 for analysis
  </Step>
  
  <Step title="Stage generation">
    AI analyzes all sources and divides the life into **3-7 stages**:
    - Age ranges (e.g., [0-17], [18-24], [25-32])
    - Era titles (e.g., "[18-24] - Startup Operator: Early Builder")
    - Summaries of what happened during each period
    - Worldview descriptions (how they thought at that stage)
    - Turning points (key moments that defined the era)
    - Confidence score (0-1, based on source coverage)
  </Step>
  
  <Step title="Source-to-stage mapping">
    For every source, AI determines which stage it belongs to:
    - Analyzes source content vs. stage descriptions
    - Assigns a relevance score (0-1)
    - Creates links between sources and stages
  </Step>
  
  <Step title="Deep research per stage">
    For each stage, LORE does **targeted discovery**:
    - Generates 6-8 new search queries specific to that era
    - Includes queries for rare/underground content from that period
    - Executes searches and vets new sources
    - Extracts content from new sources
    - Links new sources to the stage
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - Stage generation
const stageJson = await generateJson(
  `You create biographical life stages by age-era first, then refine with 
  context shifts.
  
  Output strict JSON object with key 'stages'. Each stage:
  - order: number
  - title: string (MUST look like '[18-24] - Startup Operator: Early Builder')
  - ageStart, ageEnd: numbers
  - dateStart, dateEnd: strings (e.g. "2010" or "unknown")
  - eraSummary: string (what happened during this period)
  - worldviewSummary: string (how they thought at this stage)
  - turningPoints: array of strings (key moments)
  - confidence: number (0-1)
  
  Use 3-7 stages.`,
  JSON.stringify({ person: person.name, sources: sourceDigest })
);

const parsed = parseJson(stageJson);
const stages = parsed.stages.slice(0, 7);
```

### Output

**3-7 stage objects**, each containing:
- Order (0, 1, 2...)
- Title with age bracket
- Age and date ranges
- Era summary (paragraph)
- Worldview summary (paragraph)
- Turning points (array of 2-6 items)
- Confidence score

**Source links** mapping each source to its most relevant stage

<Warning>
**Fallback Stages:** If AI fails to generate stages (e.g., insufficient sources), LORE creates 3 generic stages: Formative Years (0-17), Early Builder (18-29), Institutional Influence (30-45).
</Warning>

## Phase 4: Embed

**Goal:** Index all text content for semantic search, enabling intelligent retrieval for chat

### What Happens

<Steps>
  <Step title="Text chunking">
    For each source:
    - Combine rawText and transcriptText
    - Sanitize (remove broken unicode, control characters)
    - Split into chunks of up to 1,200 characters
    - Keep first 10 chunks per source
  </Step>
  
  <Step title="Batch embedding">
    Chunks are sent to OpenAI's **text-embedding-3-small** model in batches of 20:
    - Each chunk becomes a 1,536-dimension vector
    - Vectors capture semantic meaning of the text
    - Enables cosine similarity search later
  </Step>
  
  <Step title="Store with metadata">
    Each chunk is stored in the database with:
    - `personId` - Which person this chunk is about
    - `sourceId` - Which source it came from
    - `stageId` - Which stage/era it belongs to
    - `text` - The raw chunk text
    - `embedding` - The 1,536-dimension vector
    - `citation` - JSON with source title, URL, publishedAt
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - Chunking and embedding
function splitChunks(text: string, maxChars = 1200): string[] {
  const clean = sanitizeText(text).replace(/\s+/g, " ").trim();
  const chunks: string[] = [];
  let cursor = 0;
  while (cursor < clean.length) {
    const end = Math.min(clean.length, cursor + maxChars);
    chunks.push(clean.slice(cursor, end));
    cursor = end;
  }
  return chunks;
}

// Embed in batches
const chunks = splitChunks(text).slice(0, 10);
for (let i = 0; i < chunks.length; i += 20) {
  const batchTexts = chunks.slice(i, i + 20);
  const embeddings = await ctx.runAction(api.llm.embedTextBatch, 
    { texts: batchTexts }
  );
  for (let j = 0; j < batchTexts.length; j++) {
    await ctx.runMutation(internal.pipeline.insertChunk, {
      personId,
      sourceId,
      stageId,
      text: batchTexts[j],
      embedding: embeddings[j],
      citation: JSON.stringify({ sourceId, title, url, publishedAt }),
    });
  }
}
```

### Output

**Hundreds to thousands of chunks** (depending on source count), each:
- Indexed by person and stage
- Linked to original source
- Semantically searchable via embedding vector

<Info>
**Why Chunking?** Long texts exceed embedding model limits and dilute semantic meaning. Chunking ensures each embedding captures a focused topic or idea.
</Info>

## Phase 5: Publish

**Goal:** Build the visual timeline with images and finalize for user interaction

### What Happens

<Steps>
  <Step title="Image search per stage">
    For each stage, LORE searches for **person photos from that era**:
    - Uses Exa image search with queries like "Sam Altman photo Early Builder"
    - Returns 3 high-quality images per stage
    - Prefers photos that match the era's time period
  </Step>
  
  <Step title="Timeline card generation">
    LORE creates structured "timeline cards" for each stage:
    - Moment cards (major events)
    - Quote cards (notable statements)
    - Turning point cards (key life changes)
    - Media cards (videos, interviews)
  </Step>
  
  <Step title="Mark as ready">
    The person record is updated:
    - Status changes from "processing" to **"ready"**
    - Timeline becomes visible to users
    - All phases show 100% completion
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - Image search
async function searchPersonImages(
  name: string, 
  stageTitle?: string
): Promise<string[]> {
  const query = stageTitle
    ? `${name} photo ${stageTitle.replace(/^\[.*?\]\s*-\s*/, "")}`
    : `${name} photo portrait`;

  const response = await fetch("https://api.exa.ai/search", {
    method: "POST",
    headers: { "x-api-key": EXA_API_KEY },
    body: JSON.stringify({ query, numResults: 5, type: "auto" }),
  });
  
  const data = await response.json();
  const images = data.results
    .map(r => r.image)
    .filter(img => img?.startsWith("http"));
  
  return [...new Set(images)].slice(0, 3);
}
```

### Output

A **fully interactive timeline** with:
- Stage chapters with images
- Timeline cards for each era
- Embedded image galleries
- Ready-to-use chat panels
- Source citations for all content

## Chat: How Retrieval Works

When a user opens chat and asks a question, LORE uses the embedded chunks for intelligent retrieval:

<Steps>
  <Step title="Embed the question">
    User's question is embedded into a 1,536-dimension vector using the same model.
  </Step>
  
  <Step title="Semantic search">
    LORE computes **cosine similarity** between the question vector and all chunk vectors for that stage:
    - Chunks are ranked by similarity score (0-1)
    - Top 8 most relevant chunks are selected
  </Step>
  
  <Step title="Fallback retrieval">
    If fewer than 3-4 chunks are found for the stage, LORE expands search:
    - Searches across ALL stages for this person
    - User sees a "Cross-stage retrieval" banner
  </Step>
  
  <Step title="Generate response">
    Top chunks are passed to GPT-4 with:
    - System prompt: "You are [person] during [stage]. Answer using only evidence from these sources."
    - User question
    - Retrieved chunks as context
    - GPT-4 generates a grounded, evidence-based response
  </Step>
  
  <Step title="Return with citations">
    Response includes:
    - Answer text
    - Array of source citations (title + URL)
    - Fallback indicator if cross-stage retrieval was used
  </Step>
</Steps>

### Code Example

```typescript
// From convex/pipeline.ts - Semantic search
export const scoreAndSelectChunks = action({
  handler: async (ctx, args) => {
    const topK = args.topK ?? 8;
    
    // Embed the user's question
    const qEmbedding = await ctx.runAction(api.llm.embedText, 
      { text: args.query }
    );
    
    // Get all chunks for this stage
    const stageChunks = await ctx.runQuery(
      internal.pipeline.listChunksByStage,
      { stageId: args.stageId }
    );
    
    // Score by cosine similarity
    const scored = stageChunks
      .map(chunk => ({
        chunk,
        score: cosineSimilarity(qEmbedding, chunk.embedding),
      }))
      .sort((a, b) => b.score - a.score)
      .slice(0, topK);
    
    // Fallback if insufficient results
    if (scored.length < 3) {
      const allChunks = await ctx.runQuery(
        internal.pipeline.listChunksByPerson,
        { personId: args.personId }
      );
      return {
        usedFallback: true,
        chunks: allChunks
          .map(chunk => ({ chunk, score: cosineSimilarity(...) }))
          .sort((a, b) => b.score - a.score)
          .slice(0, topK),
      };
    }
    
    return { usedFallback: false, chunks: scored };
  },
});
```

<Tip>
**Evidence-Only:** The system prompt explicitly instructs GPT-4 to never fabricate information. If the retrieved chunks don't contain an answer, the AI says "I don't have enough information" rather than guessing.
</Tip>

## Error Handling & Resume

LORE is designed to handle failures gracefully:

### Phase Completion Tracking

Each phase updates a `jobs` table record:
- `status`: "queued" → "running" → "done" or "failed"
- `progress`: 0-100%
- `error`: Error message if failed

### Resume Logic

If a phase fails:
1. User sees which phase failed (e.g., "Failed during extract")
2. User can click **"Resume from extract"**
3. Pipeline restarts at that phase, preserving all prior work

Example: If Extract fails at 60% (30 of 50 sources extracted), resume will:
- Skip the 30 sources that already have text
- Only extract the remaining 20 sources
- Continue to Stage phase

### Retry from Scratch

User can also click **"Retry from scratch"** to:
- Delete all existing sources, stages, chunks
- Start fresh at Discover phase
- Generate new search queries

<Warning>
**Convex Function Timeouts:** Each action has a timeout (typically 5-10 minutes). Long-running phases like Extract automatically save progress incrementally to avoid losing work.
</Warning>

## Performance & Scalability

### Typical Timelines

- **Discover:** 20-40 seconds
- **Extract:** 1-3 minutes (depends on source count)
- **Stage:** 1-2 minutes (includes deep research)
- **Embed:** 30-60 seconds (depends on chunk count)
- **Publish:** 10-20 seconds

**Total:** 2-5 minutes for most timelines

### Resource Usage

- **Exa API calls:** 8-12 per Discover, 6-8 per stage (40-80 total)
- **OpenAI API calls:**
  - GPT-4: 3-5 per phase (15-30 total)
  - Embeddings: 1 call per 20 chunks (50-500 calls)
- **Database records:**
  - 1 person
  - 20-100 sources
  - 3-7 stages
  - 200-2000 chunks
  - 5-20 jobs

<Info>
**Real-Time Updates:** Convex provides instant updates to the UI as each phase progresses. Users see the progress bar update without refreshing the page.
</Info>
